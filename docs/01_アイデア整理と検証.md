# アイデア整理と検証

これまでのアイデアメモ（2026-05）を整理し、
数学的に正しい部分・要注意な部分・要証明の部分を仕分けする。

## 0. 前提の確認 — 四色定理の現在地

目標設定を誤らないため、既知の事実を先に押さえる。

- 四色定理は **証明済み**。Appel–Haken (1976) が約1500個の「可約配置」をコンピュータで検証して証明。
  Robertson–Sanders–Seymour–Thomas (RSST, 1996) が配置を633個に整理して簡略化。
  Gonthier (2005) が証明全体を Coq で形式検証。
- ただし **人間が紙の上で追える証明は今も存在しない**（すべて計算機支援）。
- アルゴリズム面では、RSST により **任意の平面地図を O(n²) で4彩色する算法** が既に存在する。
  5彩色なら O(n) で簡単（Kempe 鎖だけで証明・構成できる）。

したがって本プロジェクトの「解く」には3つのレベルがある：

| レベル | 内容 | 難易度 |
|---|---|---|
| A | 計算機に頼らない新しい証明（の手がかり） | 歴史的難問。長期目標 |
| B | 自分のアイデアに基づく4彩色アルゴリズムの構成 | 現実的。本体 |
| C | ヘックス表現での実験・観察から構造を探す | すぐ着手可能 |

**方針：C → B を実装で進めながら、A の手がかりを探す。**

## 1. 用語の統一（グロッサリ）

呼び名が揺れやすいので統一する。

| 統一名 | 別名 | 意味 |
|---|---|---|
| ヘックス盤 | ハニカム | 六角形セルを三角形（または台形）に敷き詰めた盤面 |
| セル | マス、ノード | 盤面上の六角形1個。コードでは P(n, k)（0始まり） |
| クラスタ | エリア、国、同色ノード群 | 元の地図の1つの国に対応する、連結なセルの集合 |
| 内部エッジ | same_value=True のエッジ | 同一クラスタ内の隣接セル対（同色強制） |
| 境界エッジ | same_value=False のエッジ | 異なるクラスタ間の隣接セル対（異色強制） |
| 三枝地図 | cubic map | どの地点でも国境線が3本しか集まらない地図 |
| 格子点 | — | 六角形の頂点。ハニカム格子の頂点は次数3 |

座標系は **P(n, k)（0始まり、n=行、k=行内位置）** に統一する。

## 2. 検証

### 2.1 ハニカム変換（アイデア1）

**✅ 正しい部分：三枝地図への帰着**

「四色定理は三枝地図についてだけ示せばよい」（Cayley 1879）は古典的で健全。
理由：4カ国以上が1点に集まる場所に小さな新しい国（円盤）を置けば三枝地図になり、
その地図が4彩色できれば、小さい国を点に縮めても彩色は壊れない
（点で接するだけの国同士は同色でよいから）。
→ **「無色ノード」のアイデアはまさにこの古典的トリックに対応する**。健全。

ヘックス盤は「どの格子点でも3つのセルしか接しない」ので三枝地図の一種。
ここまでは方向性として筋が通っている。

**⚠️ 要修正：「隣接数が3以下なら4色で塗れるはず」には2つのギャップがある**

1. **隣接数3なのはセルではなく格子点。**
   ハニカム格子で次数3なのは六角形の「頂点」。
   六角形の **セル同士の隣接数は最大6**（セル隣接グラフは三角格子）。
   ちなみにクラスタ制約なしのヘックス盤セルは周期パターンで3色で塗れる。
2. **クラスタ制約が次数を復活させる。**
   「最大次数 Δ のグラフは Δ+1 色で貪欲彩色できる」という定理は、
   ノードが独立に色を選べるときの話。
   1つのセルに色を置くとそのクラスタ全体の色が確定するため、
   実効的な「隣接数」はクラスタ単位の隣接数になり、これは元の地図の国の隣接数そのもの
   （＝いくらでも大きくなりうる）。

**結論：ハニカム変換は問題を「易しくする」変換ではなく「正規化する」変換。**
難しさ（クラスタ間の大域的な絡み合い）はそのまま残る。
ただし正規化自体に大きな価値がある：

- 座標が付き、走査順（インデックス）が定義でき、データ構造が単純になる（実装に直結）
- すべての地図が同一の土俵に乗るので、実験・統計・パターン観察がやりやすい
- 既知の同値な定式化と接続する：**Tait の定理 (1880)** —
  「四色定理 ⇔ 橋のない3正則平面グラフは3辺彩色可能」。
  ヘックス盤の格子点グラフは3正則なので、辺の3彩色という別視点が同じ盤面上で使える。

**📋 要証明（プロジェクトの証明課題その1）：ヘックス化可能性**

「任意の地図はトポロジー的に等価なヘックス盤上のクラスタ地図に変換できる」。
直感的には正しそうだが、きちんと示すべきことは：

- (a) 元の地図で接していた国同士は、変換後も少なくとも1つの境界エッジを共有する（隣接を失わない）
- (b) 元の地図で接していなかった国同士が、変換後に境界エッジを持ってしまわない（隣接を作らない）
- (c) 盤のサイズ m は地図の複雑さに対してどれくらい必要か（多項式で抑えたい）

グラフ描画理論に近い結果（平面グラフの直交格子描画、最大次数3の平面グラフの
六角格子描画など）があるはずなので文献調査の価値あり。**(b) で「点接触」を
分離するために無色ノード（=小領域）が必要になる**、というのがアイデア2の指摘と合致する。

### 2.2 エッジ属性と衝突修正（アイデア2）

**✅ 正しい部分：same_value 定式化と Union-Find 縮約**

「Union-Find で同色制約を縮約 → 矛盾検査 → 異色制約グラフを彩色」という手順は
正しい。本プロジェクトのデータ構造の基礎として採用する。

**⚠️ ただし難しさの所在に注意**

この手順の最後のステップ「異色制約グラフの彩色」に問題の難しさが **全部** 残っている。

- 一般のグラフでは k 彩色判定は NP 困難。
- クラスタ縮約グラフは平面グラフ（平面グラフのマイナーは平面）なので、
  四色定理により **4色解は必ず存在する**。
  つまり「存在するか」は自明で、**「どう構成するか」がプロジェクトの本体**。
- 「1クラスタ1色」案は色数を無視しているので、四色問題の文脈では意味を持たない。

**✅ 鍵となる気づき：「衝突したときの修正方法」= Kempe 鎖**

アイデア2の課題「先に塗ったノードと色が衝突したとき、どう修正するか」は、
四色問題の歴史でいう **Kempe 鎖（ケンプ鎖）** そのもの。

- 2色 a, b だけを通る連結成分（Kempe 鎖）の中で a↔b を一斉に入れ替えても彩色の正しさは保たれる。
  これが「局所的な衝突を大域的な塗り替えで解消する」古典的な道具。
- Kempe (1879) はこれで四色定理を「証明」したが、次数5の国の場合の議論に誤りがあり
  Heawood (1890) が反例を指摘。ただし同じ道具で **5色定理は正しく証明できる**。
- 素朴な「貪欲彩色＋失敗したら Kempe 鎖で修復」は **無限ループに陥る既知の例がある**
  （Errera グラフ、17頂点）。→ 実装したら最初に試すべきテストケース。

つまりアイデア2のアルゴリズム構想は「インデックス順に貪欲彩色し、衝突したら Kempe 鎖で修復する」
という形に定式化でき、**どこで・なぜ修復が失敗しうるかをヘックス盤上で観察する**ことが
レベルC（実験）の中心課題になる。

### 2.3 落とし穴チェックリスト（自己検証用）

今後「うまくいきそう」に見えたとき、必ずここに照らす。

1. **平面グラフの3彩色判定は NP 完全。**
   もし考えたアルゴリズムが「色数を選ばず局所規則だけで動く」なら、
   3色にも適用できてしまわないか確認する。できてしまうなら、どこかで誤っているか、
   平面性＋4色という条件を本質的に使えていない。
2. **4彩色の存在自体は（四色定理により）常に Yes。**
   「解けた」と言うには、4CT を仮定せずに必ず停止して彩色を返すこと、
   あるいは 4CT を仮定した上で既知（O(n²)）より良い計算量・単純さを示すこと。
3. **次数論法はクラスタ制約の下では使えない**（2.1 のギャップ2）。
4. **小さい盤・少ないクラスタでの成功は証拠にならない。**
   ランダム平面地図はほとんどの場合簡単に塗れる。難しい例（Errera 等の既知の
   反例グラフをヘックス化したもの）でテストする。

## 3. このプロジェクト固有の面白い問い

検証を踏まえて、オリジナリティがありかつ着手可能な問いを挙げる。

- **Q1.** ヘックス盤＋走査順（行順／スパイラル順）で貪欲彩色したとき、Kempe 修復が必要になる場面は
  盤面上でどんな形をしているか？ 走査順をうまく選ぶと修復が常に局所で済むか？
- **Q2.** ヘックス化に必要な盤サイズ m は国数 n に対してどう増えるか？
- **Q3.** Tait の3辺彩色をヘックス盤の格子点グラフ上で考えると、
  クラスタ境界エッジ／内部エッジの区別はどう翻訳されるか？
- **Q4.** Errera グラフ等の「Kempe 鎖が失敗する地図」はヘックス盤上でどう見えるか？
  ヘックス座標が与える追加構造（行・列・走査順）は失敗の回避に使えるか？
