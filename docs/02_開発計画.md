# 開発計画

[01_アイデア整理と検証.md](01_アイデア整理と検証.md) の方針（C: 実験 → B: アルゴリズム構成 → A: 証明の手がかり）に沿ったロードマップ。

## リポジトリ構成

```
四色定理/
├── README.md
├── docs/                  … 整理・検証ドキュメント（このフォルダ）
├── fourcolor/             … Python パッケージ（標準ライブラリのみで動作）
│   ├── dsu.py             … Union-Find（same_value=True の縮約用）
│   ├── hexgrid.py         … 三角形ヘックス盤 P(n, k) の座標・隣接（初期モデル）
│   ├── mapgen.py          … ランダムなクラスタ地図の生成（テスト用地図の無限供給源）
│   ├── verify.py          … 彩色の検証器（same_value 制約をすべて満たすか）
│   └── baseline.py        … バックトラック4彩色（正解保証つきの基準実装＝オラクル）
├── tests/                 … ユニットテスト（python -m unittest）
└── demo.py                … ランダム地図を生成 → 彩色 → 盤面表示
```

設計上のポイント：

- **検証器（verify）を最初に作る。** 以後どんなアルゴリズムを試しても、出力の正しさは
  機械的にチェックできる。「自分のアルゴリズム」と「検証器」を分離するのが実験の生命線。
- **baseline はオラクル。** 遅くても確実なバックトラック彩色を持っておき、
  新アルゴリズムの答え合わせ（彩色可能なのに失敗していないか）に使う。
- **ヘックス盤上でクラスタをランダム生成すれば、ヘックス化変換を作る前から
  実験を始められる**（隣接セルを確率的に併合すれば連結なクラスタができ、
  縮約グラフは自動的に平面グラフになる）。Phase 3 の変換器が完成するまでの足場。

## 実験の規律

- すべての彩色結果は `verify.violations()` を通す（自己申告の「成功」を信じない）
- 乱数 seed は必ず記録し、失敗例は `tests/` に固定ケースとして追加する
- 「うまくいった」と思ったら docs/01 §3.3 の落とし穴チェックリストに照らす

## 環境

- Python 3.13（標準ライブラリのみ。networkx 等は必要になった時点で導入）
- テスト実行：`python -m unittest discover -s tests -v`
- デモ実行：`python demo.py [盤サイズ] [seed]`
