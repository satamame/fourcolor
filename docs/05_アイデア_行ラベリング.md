# アイデア: 行ラベリング（行単位の仮ラベル付け）

エッジの same_value だけを入力として、各ノードに「色に関する情報」を
段階的にセットしていくアルゴリズムの第一歩。
まず行単位（r が同じ一連のノード）で仮ラベルを付け、
そのあと隣接する行同士を調整する、という二段構えの作戦。

docs/02 の Phase 2（走査順の定義 → 貪欲彩色 → 衝突修正）の
「走査順の定義」と「データの正規化」にあたる。

## 1. アルゴリズム（行内の仮ラベル付け）

行 r を p = 1, 2, ... の順に走査し、各ノードに仮ラベル (r, c) をセットする。

1. ノード (r, 1) には仮ラベル **(r, 1)** をセットする。
2. ノード (r, p)（p ≥ 2）では、ひとつ前のノードとのエッジを見る:
   - **same_value = True** … 前のノードと**同じ**仮ラベルをセットする
   - **same_value = False** … 2個目の要素を**+1**した仮ラベルをセットする
   - **same_value = None**（前のノードが無色ノード）…
     その行で**最後にセットした仮ラベルの2個目の要素に +2** した仮ラベルをセットする
3. **無色ノードには仮ラベルをセットしない。**

端のケースの定義:

- 行の先頭 (r, 1) が無色ノードの場合は、その行で最初に現れる
  非無色ノードに (r, 1) をセットして以降は同じ規則を使う。
- 無色ノードが2個以上連続しても、「最後にセットした値の +2」なので扱いは同じ
  （ギャップは2のまま増えない）。

## 2. 具体例

サンプル01（4カ国）の行3 `B B D C C`:

| ノード | (3,1) | (3,2) | (3,3) | (3,4) | (3,5) |
|---|---|---|---|---|---|
| 国 | B | B | D | C | C |
| 仮ラベル | (3,1) | (3,1) | (3,2) | (3,3) | (3,3) |

サンプル03（7カ国＋無色ノード）の行4 `E E E X C B B`（X は無色ノード）:

| ノード | (4,1) | (4,2) | (4,3) | (4,4) | (4,5) | (4,6) | (4,7) |
|---|---|---|---|---|---|---|---|
| 国 | E | E | E | X | C | B | B |
| 仮ラベル | (4,1) | (4,1) | (4,1) | — | (4,3) | (4,4) | (4,4) |

- (4,3) と (4,4) は c が1違い → C と B はこの場所で国境を接している
- (4,1) と (4,3) は c が2違い → E と C の間にこの場所では制約がない

## 3. この仮ラベルが持つ性質

- **(r, c) は「行 r の c 番目のセグメント（国の断片）」の識別子になる。**
  行内の同色強制（True エッジ）はラベルの共有として処理済みになる。
- **c が1違いのセグメント同士は、最終的に違う色にする必要がある**
  （間に False エッジがあるから）。
- **c が2違い（無色ノードを挟むギャップ）は「この場所からは制約が生じない」を表す。**
- 1個目の要素 r により行ごとに名前空間が分かれるので、
  各行を**完全に独立に**処理できる。
- 仮ラベルの辞書式順序 (r, c) が、そのまま彩色の**走査順（インデックス）**になる。

## 4. 注意点（設計時に忘れないこと）

1. **(r, c) は「国」ではなく「セグメント」の ID。**
   U 字型の国は同じ行に2回現れる（例: 行が `B B A A B B` なら (r,1) と (r,3) は同じ国）。
   これは行内では分からず、行間調整で縦エッジの True を辿って併合したとき初めて判明する。
   つまり行間調整は「色の調整」の前に「同一国の併合（Union-Find）」を含む。
   なお併合の結果 c が1違いのセグメントが同じ国になることは、
   入力が正しい地図なら起こらない（国が自分自身と国境を接することになるため）。
2. **ギャップ2は「ここでは制約なし」であって「同色にしてよい」ではない。**
   無色ノードを挟んだ2国が、盤面の別の場所で本物の国境を接しているかもしれない。
   制約の全体は最終的に「全 False エッジの集合」で持つ。
3. このアルゴリズムが完成して得られるのは、baseline.py の Union-Find 縮約と同じ
   「国の隣接グラフ」である。**彩色の難しさ自体は行間調整のステップに丸ごと残る**
   （docs/01 §2.1 の「難しさはクラスタ単位の絡み合いに残る」がここで再登場する）。

## 5. それでもこのアプローチに価値がある理由

- セグメントが**区間**という強い構造を持つ（行は1次元なので扱いやすい）
- (r, c) が走査順を与え、「盤面のどの行のどの区間で何が起きたか」を追える形になる。
  Kempe 鎖修復を観察するとき、baseline の Union-Find には無い可視性が得られる
- 行という単位で処理が進むので、アルゴリズムを少しずつ作って少しずつ検証できる

## 6. 行間調整フェーズの素描（次に設計するもの）

行 r と r+1 の間のエッジは ▲(r, p)（p 奇数）と ▽(r+1, p+1) の間にだけある。これを使い:

1. **併合**: 縦エッジが True なら、両端のセグメントは同じ国 → Union-Find で併合
2. **制約収集**: 縦エッジが False なら、両端のセグメント（の代表元）間に異色制約を追加
3. 縦エッジが None なら何もしない

ここまでで得られる「セグメント併合後の国＋異色制約」が、
baseline.py の build_cluster_graph と一致するはず。

**最初のテスト**: 3つのサンプル（01, 02, 03）すべてで、
行ラベリング＋行間併合の結果が baseline の縮約結果と一致することを機械検証する。

その土台ができてから、本丸の「行間で色をどう調整するか（貪欲彩色＋Kempe 鎖修復）」に進む。
